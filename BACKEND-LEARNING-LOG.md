## 一、这个后端项目目前在做什么

当前这个后端项目的目标，不是直接做一个复杂的大型系统，而是先做一个“我能真正看懂、运行、维护”的最小后端版本。

这个版本要解决的核心问题是：

1. 我不想每次都手改 `downloads.html` 和 `private.html`
2. 我希望把资料信息统一保存到数据库里
3. 我希望前端页面能自动读取这些数据
4. 我希望公开资料和私人资料能够分开显示
5. 我希望以后新增资料时，有一个属于我自己的管理入口

所以，这一版后端的本质是：

- 我自己写一个本地后端服务
- 我自己维护一个本地数据库
- 我自己提供 API 给前端页面使用
- 我自己做一个本地管理页来录入资料

这就是一个非常适合后端入门者的第一个完整项目。

## 二、这次后端版本的整体架构

这次我们做出来的结构可以这样理解：

### 1. 前端页面

- `index.html`：主页
- `downloads.html`：公开资料页
- `private.html`：私人资料页
- `manage.html`：本地资料管理页

### 2. 前端脚本

- `downloads.js`：从后端 API 读取公开资料
- `private.js`：密码正确后，从后端 API 读取私人资料
- `manage.js`：新增、查看、删除资料

### 3. 后端核心

- `backend/server.js`：启动本地服务器，提供 API 和静态页面
- `backend/database.js`：负责操作 SQLite 数据库
- `backend/init-db.js`：初始化数据库

### 4. 数据库存储

- `backend/data/resources.db`

这个数据库里保存的不是大文件本身，而是：

- 资料标题
- 资料说明
- 资料分类
- 公开还是私人
- 阿里云盘链接
- 阿里云盘提取信息
- 百度网盘链接
- 百度网盘提取码
- 更新时间

也就是说：

- 文件本体继续放阿里云盘 / 百度网盘
- 数据库只保存“资料的描述信息和入口”

这样的设计比较轻，也比较适合当前阶段。

## 三、为什么这次这样做

### 1. 为什么不用复杂数据库软件

因为我刚开始学后端，现在最重要的是把流程跑通，而不是一开始就陷进复杂安装和运维。

所以这次选择：

- 使用 Node 自带的 `SQLite`

这样做的好处：

- 不需要额外安装 MySQL、PostgreSQL
- 不需要配置数据库账号密码
- 不需要额外往 `C 盘` 安装东西
- 数据库文件就在项目目录里，方便理解和管理

### 2. 为什么用本地后端

因为我现在的目标是先学明白：

- 后端怎么启动
- API 是什么
- 数据怎么进数据库
- 前端怎么从后端拿数据

本地后端最适合练这些基础能力。

### 3. 为什么要做 `manage.html`

因为如果没有管理页，我还是要手动改代码。

而做了管理页之后，我真正体验到的是：

- 后端管理数据
- 前端自动展示

这才是“动态网站”的核心感觉。

### 4. 为什么公开资料和私人资料分开

因为业务上它们就不一样：

- `public`：公开资料，显示在 `downloads.html`
- `private`：私人资料，显示在 `private.html`

这让数据库字段设计、API 调用和前端显示逻辑都更清晰。

## 八、这次遇到的问题与修正

### 问题 1：PowerShell here-string 命令第一次写错

报错核心是：

```text
No characters are allowed after a here-string header but before the end of the line.
```

原因：

- PowerShell 的 here-string 写法要求很严格
- 我在 `@'` 后面混入了不该有的字符

修正方法：

- 把长命令拆开
- 分开写文件
- 单独创建 `.gitkeep`

我学到的点：

- 批量写文件时，不要为了省一步把命令写得过长
- 稳定比一条命令写到底更重要

### 问题 2：文档一次性写太多，命令超时

现象：

- 一次性写 `README.md`、`HANDOFF.md`、`BUILD-LOG.md`、`ERROR-LOG.md` 时超时

原因：

- 单次命令内容太长
- 不是文档内容有问题，而是执行时间不够

修正方法：

- 分成多次写入
- 先写两个文件，再写另外两个

我学到的点：

- 命令执行超时不一定是功能错误
- 有时候只是“任务太大，要拆开做”

## 十、如果面试官问我，我可以怎么讲

### 1. 这个项目是做什么的

我可以这样回答：

> 我做了一个个人资料管理的小型全栈项目。前端负责展示公开资料和私人资料，后端用 Node 提供 API，SQLite 保存资料元数据，管理页用于新增和删除资料。这样我不需要每次手改网页，而是通过数据库维护内容。

### 2. 为什么选 SQLite

我可以这样回答：

> 因为这是我的第一个后端练习项目，我希望先把后端、数据库、API、前端联调的完整流程跑通。SQLite 足够轻量，不需要额外安装数据库服务，适合快速验证数据建模和接口设计。

### 3. 后端做了哪些 API

我可以这样回答：

> 我做了健康检查接口、资料列表查询接口、资料新增接口和资料删除接口。前端页面分别根据公开和私人可见性去调用不同数据。

### 4. 为什么有 `public` 和 `private`

我可以这样回答：

> 因为资料有公开和私人两种展示场景，所以我把这个业务差异直接设计进数据库字段 `visibility`，这样前端筛选和后端查询都会更清晰。

### 5. 这个项目最大的收获是什么

我可以这样回答：

> 最大的收获是第一次真正把“数据库、后端 API、前端展示、管理后台、错误处理、文档记录”串成了一个完整的小项目，而不是只停留在写单个页面。

## 十二、后端学习阶段的固定约定

- `BACKEND-LEARNING-LOG.md`：记录详细学习过程
